# Doc Forge 本番運用移行計画（2026-03）

## 0. 結論（先に決める）

最終運用は **Webアプリ運用を主軸** にする。
ただし、機密要件が強い環境向けに **ローカル実行モードを補助的に残す**。

- 標準: Cloud Web（シングルテナントから開始）
- 補助: Local Mode（オフライン/閉域環境向け）

この構成にすると、以下を同時に満たせる。

- チーム利用・複数端末利用・継続的改善は Web で回せる
- 高機密案件は Local Mode で切り離せる
- プロダクト本体は1つのコードベースで進化できる

---

## 1. なぜ Web 主軸か（Local 専用にしない理由）

### Web 主軸の利点

- セッション/履歴/設定を中央管理できる
- 認証・監査ログ・バックアップを運用として成立させやすい
- デプロイで全ユーザーへ同時に改善を届けられる
- 利用状況と障害を監視できる（SLO運用が可能）

### Local 専用の弱点

- 端末単位でデータが分断される
- 障害時の復旧品質がユーザー依存になる
- バージョン差分で不具合再現が難しくなる
- チームでの運用ルール統制が難しい

### Local を残す理由

- オフライン環境や閉域網での実行要件
- 外部クラウド持ち出し不可データへの対応
- 実験・検証の高速ループ

---

## 2. 運用モデル比較

| モデル | 想定用途 | 強み | 弱み | 推奨度 |
|---|---|---|---|---|
| Local Only | 個人利用、PoC | 立ち上げ最速、データが端末内 | 監視/復旧/統制が弱い | 低（本番主軸には不向き） |
| Self-host Web（単一組織） | 社内導入初期 | 認証・監査・集中管理が可能 | 運用負荷は自前 | 高（初期本番に最適） |
| Managed Cloud Web（SaaS） | 複数組織展開 | スケールしやすい、提供速度が高い | マルチテナント設計が必要 | 中〜高（中長期） |

---

## 3. ターゲット運用像（推奨）

### フェーズ終着点（1〜2四半期）

- 提供形態: **Self-host Web（単一組織）**
- 認証: OIDC/SAML ベースの SSO
- データ: PostgreSQL（マネージド）
- 秘匿情報: Secret Manager + KMS
- 監視: 構造化ログ + APM + エラートラッキング
- バックアップ: 自動スナップショット + 復旧訓練

### その先（SaaS化する場合）

- マルチテナント分離（organization_id）
- テナント単位の利用上限・監査・請求
- データ保持ポリシーをテナント別に制御

---

## 4. 実装ロードマップ

## Phase P0: 本番化前ハードニング（2〜4週間）

目的: 現行MVPを「公開して壊れない最低限」に引き上げる。

### 必須実装

1. 認証/認可
- APIを認証必須化
- セッション/メッセージ/スナップショットをユーザー境界で分離

2. 入力バリデーション
- API request body を schema 化（title/message/provider）
- 文字数上限、空文字、型不正、未知フィールドを拒否

3. レート制御と濫用対策
- `/api/chat` と設定APIに user/IP 単位レート制限
- 大量送信・長文連投の防止

4. 監査ログの最小実装
- 認証、設定変更、削除操作の監査イベント化
- 個人情報/鍵をログに出さない

5. CI ゲート
- `lint` + `build` + `e2e(mock)` を必須化
- main ブランチ保護

### 完了条件

- 未認証アクセスで主要APIが拒否される
- 不正入力で API が安全に 4xx を返す
- 負荷試験でレート制限が機能する

---

## Phase P1: Web 本番ローンチ（4〜8週間）

目的: 1組織内で継続運用できる体制を作る。

### 必須実装

1. データ基盤移行
- SQLite から PostgreSQL へ移行
- マイグレーション管理とロールバック手順を確立

2. 秘匿情報管理
- LLMキー暗号化鍵をファイル依存から Secret Manager へ移行
- 鍵ローテーション手順を文書化

3. 可観測性
- エラー監視（例: Sentry）
- API latency / LLM success rate / fallback rate をメトリクス化

4. 運用Runbook
- 障害レベル定義（SEV1〜SEV3）
- 復旧手順、連絡手順、暫定回避手順

### 完了条件

- 1か月の安定運用（重大障害なし）
- バックアップからの復旧訓練を実施済み

---

## Phase P2: 拡張運用（8〜12週間）

目的: 規模拡大・SaaS化に耐える設計に進める。

### 実装候補

- 非同期ジョブ化（長時間LLM処理、再実行制御）
- organization 境界の明示化（将来のマルチテナント化）
- 監査エクスポート、データ保持期間ポリシー
- コスト統制（モデル別上限、予算アラート）

### 完了条件

- 同時利用増加時でもSLOを維持
- テナント/組織単位での運用管理が成立

---

## 5. Local Mode の位置づけ

Local Mode は「本番の代替」ではなく、次の限定用途にする。

- 高機密案件（クラウド持ち出し不可）
- 開発・検証・デモ
- 一時的なオフライン運用

推奨運用ルール:

- Local の成果物は必要時のみ Web へインポート
- Local DB はユーザー責任で暗号化・バックアップ
- サポート優先度は Web 運用を上位に置く

---

## 6. 直近30日の実行タスク（着手順）

1. API 認証ガード追加（全API）
2. schema バリデーション導入（request/response）
3. chat/settings の rate limit 実装
4. PostgreSQL 接続層と migration 基盤追加
5. 監視基盤（エラー + メトリクス）導入
6. 運用Runbook初版作成（障害/復旧/バックアップ）

---

## 7. 意思決定ガイド（最終確認）

以下に当てはまるほど Web 主軸が妥当:

- 3人以上で利用する
- 複数端末で継続利用する
- 監査ログ/復旧責任を求める
- 障害時の運用責任をチームで持つ

以下が主要件なら Local 優先もあり:

- 完全オフライン必須
- 端末外にデータを出せない
- 個人利用が中心

**Doc Forge の現状と将来拡張性を踏まえると、最終形は Web 主軸 + Local 補助が最も現実的。**
