# Doc Forge - コンセプトドキュメント

## 一言で

LLMとの対話を通じて、アイデア・スライド・企画書などのドキュメントをブラッシュアップ・完成させるWebアプリ。
階層的なドキュメント（スライド構成 → 各スライド）にも対応し、全体と部分を行き来しながら磨き上げる。

## なぜ必要か — 根本の課題

### 「考える」と「書く」が断絶している

人はアイデアを形にするとき、2つのことを同時にやっている：
- **思考**: 何を伝えるか、どう構成するか考える
- **執筆**: それを文章・スライドとして書き下す

現状のツールはこの2つを別々に扱う。
メモアプリで書き、ChatGPTで壁打ちし、結果をコピペして整形する。
この往復のたびに思考の流れが途切れる。

### 既存ツールの限界

| ツール | できること | できないこと |
|--------|-----------|-------------|
| ChatGPT / Claude | 対話で思考を整理 | 成果物がチャットの中に埋もれる。手動コピペが必要 |
| Notion / Google Docs | ドキュメント作成・管理 | AI支援は補助的（補完・要約）。構造的な壁打ちはできない |
| Gamma / Beautiful.ai | AIスライド自動生成 | 一発生成。対話で磨き上げるプロセスがない |
| Claude Artifacts | 対話でドキュメント生成 | セッション管理なし。階層的な構造の深掘りが苦手 |

**共通の穴**: 「対話で考えながら、その結果がリアルタイムに成果物になる」体験がない。

### Doc Forgeが埋めるギャップ

```
[思考（対話）] ←→ [成果物（ドキュメント）]
      ↑                    ↑
      └── この2つが同一画面で同期する ──┘
```

- 対話＝思考のプロセスが、そのままドキュメントに沈殿する
- 一発生成ではなく、**反復的に磨く**プロセスを支援する
- 全体構成と個別セクションを行き来できるので、**構造ごと議論できる**
- 対話履歴が残るので、「なぜこの構成にしたか」が追跡可能
- **捨てた要素も消えない** — 検討の過程・ボツ案がすべてアーカイブされる

### 想定ユースケース

1. **スライド・プレゼン作成**: 構成から各スライドの中身まで対話で組み立てる
2. **1枚企画書**: アイデアの壁打ち → 構造化 → 文書化を一気通貫で
3. **ブレスト・アイデア整理**: 散らばった考えをLLMと対話して構造化する
4. **技術ドキュメント**: 設計ドキュメントやRFCの草案を対話で練る

### このプロダクトが「やらないこと」

- 長文の原稿執筆（小説、論文） → エディタ特化ツールの領域
- リアルタイム共同編集 → Google Docs/Notionの領域（Phase 3で検討）
- ドキュメントの自動生成・一発完成 → 対話で磨くプロセスこそが価値

## コアコンセプト

**「対話がそのままドキュメントになる」**

LLMとの会話がリアルタイムにドキュメントへ反映される。
ユーザーは対話を続けるだけで、ドキュメントが洗練されていく。
チャットは流れるもの、ドキュメントは積み上がるもの — この2つを同期させる。

## ドキュメントの二層構造 — Main と Archive

### 問題意識

長期にわたるアイデア出しでは、必ず「残す要素」と「捨てる要素」が出てくる。
しかし捨てた要素にも価値がある：
- 後で「やっぱりあのアイデアが良かった」と戻りたくなる
- なぜ捨てたかの判断根拠が、意思決定の記録になる
- ボツ案の断片が、別の文脈で再利用できる

既存ツールでは、削除＝消滅。Git的な差分管理は技術者以外には敷居が高い。

### 設計: Main / Archive の分離

```
┌─────────────────────────────────────────┐
│  Main Document                          │
│  ┌───────────────────────────────────┐  │
│  │ 現在のドキュメント（見せる状態）    │  │
│  │ - 採用されたセクション             │  │
│  │ - 最新の合意内容                   │  │
│  └───────────────────────────────────┘  │
│                                         │
│  Archive                                │
│  ┌───────────────────────────────────┐  │
│  │ 検討の過程（すべて残る）           │  │
│  │ - ボツになった構成案               │  │
│  │ - 却下されたセクション・文案       │  │
│  │ - 書き換え前の旧バージョン         │  │
│  │ - 「なぜ変えたか」の対話スナップ   │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

### どう機能するか

1. **ドキュメント更新時**: 旧版は自動的にArchiveに退避
2. **セクション削除時**: Mainから消えるが、Archiveに「却下」として残る
3. **Archiveの参照**: 中央ペインで「アーカイブを見る」トグルで閲覧可能
4. **復活**: Archiveからワンクリックで要素をMainに戻せる

### UIへの影響

中央ペインに3つ目のモードが加わる：

| モード | 用途 |
|--------|------|
| アウトラインモード | 全体構成の俯瞰 |
| フォーカスモード | セクションの深掘り |
| **アーカイブモード** | 過去の検討過程・ボツ案の閲覧・復活 |

あるいは、フォーカスモード内でサイドドロワー的にそのセクションの変更履歴を見る形でもよい。

## UI構成（3ペインレイアウト）

```
┌──────────┬────────────────────────┬──────────────┐
│          │                        │              │
│ Session  │   Document Preview     │   Chat       │
│ Panel    │   (Markdown)           │   Panel      │
│          │                        │              │
│ - 一覧   │   - リアルタイム        │  - LLMと対話  │
│ - 新規   │     プレビュー          │  - 提案の     │
│ - 履歴   │   - 編集も可能          │    承認/却下  │
│          │                        │              │
└──────────┴────────────────────────┴──────────────┘
```

### 左ペイン：セッション管理
- セッション一覧（日時・タイトル）
- 新規セッション作成
- 過去セッションの閲覧・再開

### 中央ペイン：ドキュメントプレビュー
- Markdownで記述されたドキュメントのライブプレビュー
- 必要に応じてMarkdownソースの直接編集も可能
- LLMからの変更提案がハイライト表示される（diff表示）
- **2つの表示モード**:
  - **アウトラインモード**: 見出し構造を俯瞰（全体構成の議論に）
  - **フォーカスモード**: 特定セクションにズームイン（中身の深掘りに）

### 右ペイン：対話パネル
- LLMとのチャットインターフェース
- ドキュメントに対する指示・質問・フィードバック
- LLMの提案をワンクリックでドキュメントに反映
- **入力方式**: テキスト入力 + 音声入力（後述）

## ユーザーフロー

### 基本フロー
1. **セッション開始**: テーマやテンプレートを選んでスタート
2. **初期入力**: ざっくりしたアイデアやキーワードを入力
3. **対話ループ**:
   - ユーザーが指示やフィードバックをチャットで送る
   - LLMがドキュメントの更新案を生成
   - ユーザーが承認/修正/却下する
   - ドキュメントが更新される
4. **完成**: 満足したらエクスポート（Markdown / PDF / HTML）

### 階層ドキュメントのフロー（例：スライド作成）
1. **全体構成を対話で決める**（アウトラインモード）
   - 「AIの社内導入提案のスライドを作りたい」
   - LLMが見出し構成を提案 → ユーザーが調整
2. **セクション単位で深掘り**（フォーカスモード）
   - 見出しをクリックしてフォーカス
   - 対話のコンテキストがそのセクションに絞られる
   - 「この課題セクション、もっと具体的なデータが欲しい」
3. **全体に戻って整合性を確認**（アウトラインモード）
   - 流れの確認、重複の排除、トーンの統一
4. **仕上げ → エクスポート**

```
アウトライン ──→ フォーカス ──→ フォーカス ──→ アウトライン ──→ 完成
(全体構成)    (Slide 1)    (Slide 2)    (全体調整)
```

## 技術スタック

| レイヤー | 技術 |
|---------|------|
| フロントエンド | Next.js (App Router) + TypeScript |
| UIフレームワーク | Tailwind CSS + shadcn/ui |

| バックエンド | Next.js API Routes |
| LLM連携 | Anthropic Claude API (or OpenAI API) |
| データ永続化 | SQLite (ローカル) or PostgreSQL |
| 認証 | なし（MVP） → NextAuth（Phase 2） |

## 設計方針

- **ローカルファースト**: MVPはローカルストレージ/SQLiteで完結
- **ストリーミング**: LLMの応答はストリーミングでリアルタイム表示
- **シンプルなデータモデル**: Session → Messages + Document の最小構成
- **LLM非依存**: プロバイダーを差し替え可能な抽象レイヤー

## データモデル（概要）

```
Session
  ├── id
  ├── title
  ├── created_at
  ├── updated_at
  ├── document_content (Markdown)     ← Main: 現在の状態
  ├── messages[]
  │     ├── id
  │     ├── role (user | assistant)
  │     ├── content
  │     └── created_at
  └── snapshots[]                     ← Archive: 変更の履歴
        ├── id
        ├── section_heading            (どのセクションの変更か)
        ├── previous_content           (変更前の内容)
        ├── action (replaced | removed | restructured)
        ├── reason                     (変更理由。対話から自動抽出)
        ├── related_message_id         (きっかけとなった対話へのリンク)
        └── created_at
```

**ポイント**: Archiveは差分ではなくスナップショットで持つ。
差分だと復元時に面倒になるし、「あのとき何が書いてあったか」をそのまま読める方が体験として良い。

## LLMプロンプト設計

### 基本構造

LLMへの各リクエストは以下の3層で構成する。

```
┌─────────────────────────────────────────┐
│ System Prompt（固定）                     │
│   - Doc Forgeとしての振る舞いルール       │
│   - 出力フォーマットの指定               │
├─────────────────────────────────────────┤
│ Document Context（動的）                  │
│   - 現在のドキュメント全文 or セクション   │
│   - フォーカス中のセクション情報          │
├─────────────────────────────────────────┤
│ Conversation（動的）                      │
│   - 直近の対話履歴                       │
│   - ユーザーの最新メッセージ              │
└─────────────────────────────────────────┘
```

### System Prompt の設計方針

```markdown
あなたはドキュメント共同編集者です。
ユーザーとの対話を通じて、Markdownドキュメントを磨き上げます。

## 返答ルール
1. 返答は「対話パート」と「ドキュメント更新パート」に分ける
2. ドキュメントの更新がある場合、必ず <doc_update> タグで囲む
3. 更新がない会話（質問・相談）には対話パートのみで返す
4. ドキュメントの更新は該当セクションのMarkdown全文を返す

## 振る舞い
- 一方的に書き直さず、まず提案・確認する
- ユーザーの意図を汲み取り、構造化を手伝う
- 「なぜそう書くか」の根拠も対話で示す
```

### 出力フォーマット

LLMの返答を「対話」と「ドキュメント変更」に構造的に分離する。

```
[対話パート]
課題セクションに具体的なデータを入れるのは良い方向ですね。
社内アンケートの結果と、業界レポートの数字の2軸で
説得力を出すのはどうでしょうか？

<doc_update section="## 課題の整理">
## 課題の整理

- 社内アンケート（2024年実施、N=150）で73%が「資料作成に週3時間以上」と回答
- McKinsey調査: ナレッジワーカーの業務時間の28%が情報整理・文書化に費やされている
- 現状のワークフロー: ブレスト → メモ → 清書 → レビュー（平均3往復）
</doc_update>
```

**この分離の利点**:
- フロントエンドで対話表示とドキュメント更新を別々に処理できる
- ユーザーは対話を読んでから、ドキュメント変更を承認/却下できる
- `section` 属性でどこを更新するか明示 → フォーカスモードと自然に連携

### モード別のコンテキスト制御

| モード | ドキュメントコンテキスト | 対話の粒度 |
|--------|------------------------|-----------|
| アウトラインモード | ドキュメント全文 | 構成・流れ・全体の議論 |
| フォーカスモード | 全体の見出し構造 + フォーカス中セクションの全文 | セクション内容の深掘り |

フォーカスモードでも見出し構造は渡す（全体の中での位置を把握させるため）。
ただしフォーカス外セクションの本文は省略してトークンを節約する。

```
# プレゼンタイトル                    ← 見出しのみ
## 背景                              ← 見出しのみ
## 課題の整理 [FOCUS]                 ← ★ここだけ全文展開
  - 社内アンケートで73%が...
  - McKinsey調査...
  - 現状のワークフロー...
## 提案                              ← 見出しのみ
## ロードマップ                       ← 見出しのみ
```

### プロンプトの進化方針

MVP時点では上記のシンプルな構成で始め、以下を段階的に追加する：
- **テンプレートごとのシステムプロンプト**: スライド用、企画書用で振る舞いを変える
- **トーン指定**: 「フォーマル」「カジュアル」「技術寄り」等
- **フィードバック要求**: LLMから「この部分が弱いので補強しませんか？」と能動的に提案

## 音声入力 — 喋りながらドキュメントが書き変わる

### 理想の体験

キーボードで打つより、喋りながらドキュメントが進化していく方が自然。
特にブレスト段階やアイデア出しでは、思考をそのまま声にする方がテンポが良い。

### 技術的にどう実現するか

```
[マイク] → [音声認識] → [テキスト化] → [LLM] → [ドキュメント更新]
```

**音声認識の選択肢**:

| 方式 | 特徴 | 遅延 |
|------|------|------|
| Web Speech API（ブラウザ内蔵） | 無料、追加依存なし。精度はそこそこ | リアルタイム |
| Whisper API（OpenAI） | 高精度。日本語も強い | 数秒（バッチ） |
| Deepgram / AssemblyAI | ストリーミング対応。高精度 | リアルタイム |

### 発話 → LLM投入のタイミング設計

喋っている最中に逐語的にLLMに送ると、不完全な入力で無駄なAPI呼び出しが発生する。

**方針: 発話の区切りを検知してバッチで送る**

```
[喋っている...] → [沈黙1.5秒検知] → [確定テキストをLLMに送信]
                                         ↓
                                    [ドキュメント更新]
```

- 沈黙（pause）を検知してまとまった発話単位でLLMに投げる
- 発話中はリアルタイムで文字起こしテキストをチャット欄にプレビュー表示
- 確定前にユーザーが修正できる「送信前プレビュー」があると安心
- 送信は明示的にボタンで確定する方式でもよい（MVP向き）

### フェーズ分け

- **MVP**: テキスト入力のみ
- **Phase 1.5〜2**: 音声入力追加（Web Speech APIで最小実装）
- **将来**: ストリーミング音声認識 + 自動送信（ハンズフリー体験）

## 実装上の注意点・設計判断

### 1. LLMの出力パースは壊れる前提で設計する

`<doc_update>` タグでの構造化出力はLLMが100%守る保証がない。

**対策**:
- パースに失敗したら、返答全体を「対話パート」として扱い、ドキュメントは更新しない
- ユーザーには「提案を検出できませんでした。もう一度お願いします」程度の通知
- Claude APIのtool use（function calling）を使う手もある。`doc_update` をtoolとして定義すれば、構造化出力の信頼性が大幅に上がる
- **推奨**: MVP段階ではtool useベースで実装し、自由形式のパースは避ける

### 2. ストリーミングとドキュメント更新のタイミング

設計方針に「ストリーミング」とあるが、ドキュメント更新はストリーミング中に中途半端な状態で反映してはいけない。

**方針**:
- 対話パートはストリーミングでリアルタイム表示する
- ドキュメント更新はLLMの応答完了後にまとめて反映する（確定してからMainを書き換える）
- tool useを使う場合は自然にこの分離が実現できる（テキスト応答 = 対話、tool call = ドキュメント更新）

### 3. 承認/却下のUX — 毎回確認は重い

現在の設計ではLLMの提案を毎回「承認/却下」するフローだが、これが頻繁だとテンポが悪くなる。

**検討案**:
- **自動適用 + Undo**: デフォルトでドキュメントに反映し、気に入らなければUndoで戻す
  - → テンポが良い。Archiveがあるので安全
- **確認必須モード**: 大きな構造変更（セクション追加/削除/並び替え）のときだけ確認を求める
- MVPでは「自動適用 + Undo」が実装もシンプルでおすすめ。Archiveの仕組みとも自然に噛み合う

### 4. セクションのマッチング問題

`<doc_update section="## 課題の整理">` は見出しテキストが識別子になっている。
しかし見出しはLLM自身が書き換えることがある（「課題の整理」→「解決すべき課題」）。

**対策**:
- 各セクションに内部ID（`section-1`, `section-2`）を振り、見出しテキストとは別に管理する
- LLMにはIDで指定させる: `<doc_update section_id="section-3">`
- フロントエンドでは見出しテキストを表示し、内部的にはIDで追跡
- これで見出しのリネームとセクションの同一性が両立する

### 5. 対話履歴のコンテキスト管理 — 要約 + オンデマンド検索

長いセッションでは対話が100メッセージを超えることもある。全部をコンテキストに入れるのは非効率。
一方で、Archiveにすべての変更履歴と対話が紐づいて残っているので、これを活かす。

**3層のコンテキスト設計**:

```
┌─────────────────────────────────┐
│ 常に含める                       │
│  - 現在のドキュメント（Main）     │
│  - 直近N件の対話（例: 20件）     │
├─────────────────────────────────┤
│ 要約として含める                  │
│  - それ以前の対話の自動要約       │
│  - 「背景→課題→提案の構成で      │
│    合意。データの裏付けを追加中」  │
├─────────────────────────────────┤
│ 必要時に取りに行く（検索）        │
│  - Archiveのスナップショット     │
│  - 過去の特定の対話              │
│  - 「前に却下した案を見たい」     │
│    → Archiveから該当部分を取得   │
└─────────────────────────────────┘
```

**ポイント**:
- ドキュメント自体が「議論の結論」なので、古い対話は要約で十分
- ユーザーが「前の案に戻したい」「あのとき何を話したか」と言ったら、Archiveから該当部分を検索してコンテキストに追加する
- 検索はキーワードベース（MVP）→ embedding検索（将来）
- 要約は一定メッセージ数（例: 20件）ごとにLLMで自動生成し、DBに保存

## 名前の由来

**Doc Forge** = Document + Forge（鍛冶場）
対話という炎でドキュメントを鍛え上げる、というメタファー。
